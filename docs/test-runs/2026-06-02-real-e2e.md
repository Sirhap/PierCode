## Bugs Found And Fixed

### QWEN-E2E-2026-06-02-001

Symptom: `scripts/qwen-context-e2e.mjs` failed at the post-compression prompt with:

```text
Qwen E2E fillAndSend failed: evaluated in tabId=104680339 type=string value={"ok":false,"error":"fillAndSend returned false"}
```

Root cause: the test relied only on the content-script E2E bridge for prompt submission. In a handoff Qwen tab, the bridge could fill but fail to submit, while the real UI send button remained usable.

Fix: added a page-UI fallback in `sendQwenPrompt`: set `textarea.message-input-textarea` with the native setter, dispatch `input`/`change`, then click `button.send-button` or dispatch Enter if no button is available.

## Issues And Optimizations

- Chrome Browser Use refused direct `chrome-extension://...` navigation by security policy. The run did not bypass that through the Chrome automation tool; the already-installed PierCode extension connected after opening a normal Qwen page, and project E2E scripts used the extension's own configure path.
- Qwen can answer as ordinary AI if the PierCode initialization button is not clicked first. Fresh attachment validation initially failed with a normal refusal; after clicking `🔗 初始化`, it produced a real `browser_screenshot` tool card and attachment result.
- Qwen occasionally returns `Tool browser_snapshot does not exists.` at the host-native layer before PierCode tool-card execution. The extension still extracted and executed the visible `piercode-tool` block correctly.
- One natural Qwen browser flow hit external `Allocated quota exceeded` before screenshot. A fresh initialized conversation avoided that and validated screenshot attachment.
- Existing Qwen conversations are numerous and long; fresh conversations are more reliable for E2E runs.
